Work Practices/How to File Bugs Tasks and Questions

How to File Bugs, Tasks, and Questions

This document describes how to log bugs, tasks (such as new features and/or things people need to work on), and project-based questions (ones that are not quickly answered in Slack and have impact on a project).

What is GitHub [Issues]?

First off, GitHub is a system that stores documents and keeps a revision history of those files. This is great for code and/or documents that you can share with the team. Aside from storage, it provides the infrastructure to track work items such as new tasks, product goals, bugs, and project questions. This part of GitHub is called Issues.

GitHub is divided into repositories (i.e. a project or group of files it stores) and has an Issue bucket that is associated with each repository.

Issues contain several fields:

  • Title: One line description of the Issue
  • Comment: A more verbose description of the Issue
  • Milestones: A sprint in which the Issue is set to be completed
  • Epics: A product goal that the Issue is associated with *(Zenhub feature)
  • Label: A tag that describes the type of Issue (examples: backend,question or bug)
  • Assignee: The person that has outstanding work to complete on an Issue

What is Zenhub?

Zenhub is a snazzy user interface for managing workflow of GitHub Issues via an Agile board.

Zenhub provides pieces of key functionality that do not exist in GitHub Issues by itself:

Zenhub Pipelines

Pipelines - are scrum board columns that indicate an Issue's state. Pipelines that we use include:

  • Backlog: Issues that have not been groomed

  • Ice Box: This is the "we'll get around to it" pipeline, where issues go when they've been groomed but aren't ready to be worked on yet

    • This may be because they're not a high enough priority (e.g. fiddly bug fixes) or they don't have sufficient information
  • Epics: This pipeline hosts Epic Issues which are product goals that consist of multiple smaller Issues
  • Ready: Issues here are ordered by priority (see below) and have enough detail that they can be pulled from at any time
  • Current Sprint: The team has committed to working on the Issues during their present work sprint (1-2 weeks)
  • In Progress: Issues that are actively being worked on
  • Validate (code repositories only): Issues that have PRs ready for review
  • Done: Issues that have been completed and their PRs merged into master

Priority

Issues live vertically within work pipelines. Zenhub lets you drag Issues above or below each other. An Issue's vertical position within a pipeline indicates its priority; with the highest priority issues at the top.

New Repo Setup

New GitHub repositories need to be added to the Zenhub board via the 'Repos' dropdown in the top left of the Zenhub board.

Important: Be careful not to unlink repos within the 'Repos' dropdown. This will effect the entire team and mess up the board.

Labels

Labels are used to organzie work by:

  • Team (Rainbows, Kudos, Unicorns, Nebula, Platform)
  • Focus (Frontend, Backend, UX, Content, etc.)
  • Product (HCA, Identity, Sitewide, etc.)
  • Type (Bug, Testing, Discovery, etc)

Zenhub Board

To link Zenhub to your GitHub repository and Issues:

  1. Go to Zenhub.com, sign in with Github, and authorize access to private repositories (you'll have to agree to several prompts)
  2. View your Zenhub agile board here --> https://app.zenhub.com/workspace/o/department-of-veterans-affairs/vets.gov-team
  3. To enable ZenHub features on GitHub, download the Zenhub Browser Extension

When to File Issues

The most common situations to file issues are:

  • New Features: Something new needs to be made or figured out
  • Bugs: Something is broken on a code project or with a team process
  • Question: Questions about the project that either will take some time to answer or it's important to document the answer to

Tips

New Features: What to Include

  • Concise title to summarize the new feature

  • The description should include:

    • What the new feature is
    • Why it's being done (if not obvious)
    • Sub-tasks that might need to be done to complete it
    • Any design images or documents that would support completing it
  • Label with what kind of team members are needed to complete it (example: developer, designer, etc.)

Bugs: What to Include

  • Concise title to summarize the problem

  • The description should include:

    • Extremely granular steps to recreate it
    • Expected outcome
    • Actual outcome
    • Screenshots or screen recordings
    • Environment that it occurred in (what username, browser, date/time, etc.)
  • Assign to someone capable of resolving it, or a team lead

Questions: What to Include

  • One sentence version of question as title
  • Description should include the context of the question, but also things you've already done to investigate finding an answer
  • Assign to someone capable of resolving it, or a team lead
