Accessibility and 508 Best Practices and Launch Guidelines

This document

  1. Describes the accessibility and 508 launch criteria for all vets.gov launches
  2. Provides best practices for our own manual QA and for working with the Section 508 office to evaluate vets.gov products

Launch Criteria

  1. Automated build e2e tests must excercise all functionality in the launched product
  2. Each rendered page must pass an aXe check (example). The default aXe rule sets (section-508, wcag2a, wcag2aa) should be used in all tests, except in the case of false positives. In those cases, rules may be selectively disabled (see #8110 for work-in-progress on how to best disable rules).

  3. Zoom Test

    • Set browser width to 1280px
    • Zoom using Ctrl and + or Cmd and + on Mac, until browser says 400%
    • Layouts should not scroll sideways or have content to the edges
    • WCAG: Understanding Reflow

  4. Keyboard Navigation

    • All natively focusable elements (links, form inputs, buttons, radios, checkboxes) reachable by keyboard
    • No keyboard traps
    • Predictable tabbing order
    • Ways to skip large groups of links (skip to content link)
    • WebAIM keyboard accessibility guide

  5. The flow must be possible to complete on vets.gov supported browsers with the most commonly used screenreader(s) (see for example 2017 WebAIM screenreader survey):

    • Safari: Voice Over
    • IE11: JAWS/NVDA (tested by the Section 508 office)
    • the flow must be possible using a keyboard only (no pointing device). See the WebAIM keyboard accessibility guide for keyboard strokes.
    • zoom browser to 400% and verify that the layout is still meaningful and the feature is still usable (see WCAG guide for understanding reflow)

The following additional tests are optional but may be considered:

  1. test all flows with Firefox: NVDA
  2. test for color blindness accessibility with tools like the Chrome Colorblinding plugin
  3. for designers, there is a Sketch plugin, Color Contrast Analyser to catch contrast issues early

Section 508 Office

vets.gov has a monthly phone call with the Section 508 office to discuss any products and accessibility issues that the 508 office has surfaced. At or after each meeting, we send them a new set of products to evaluate. Some of these products may already be live and are just in a backlog for a11y/508 testing, but in general we should ask the 508 office to review products in staging before we launch.

This tracking doc shows which products have been evaluated or are being evaluated, including tickets for addressing each issue reported.

The 508 Office likes to send the issues they find as a Word doc. Sometimes the issue they report is difficult for us to reproduce, or is ambiguous--in those cases it is the responsibility of the sprint team to contact the 508 office via email or phone call (contact info below) to clarify the issue. This can be a product manager, a designer, or an engineer. It is not OK to simply comment in the issue that it was not reproducible without following up with the 508 office to fully understand the problem.

Once the issue is understood, it should be converted to a github ticket, labeled with the product or feature that pertains to it, and assigned to the vets.gov sprint team who owns that product. Please see below section on Documenting issues found via manual QA.

Documenting issues found via manual QA

Creating actionable a11y/508 tickets is a bit of an art. If the issue is reported by the Section 508 issue, it is likely in a Word doc, and may lump together several different issues found in different products or in our headers/nav/footers. These steps might help you get started until you come up with your own system:

  1. Try to reproduce the issue using Safari and Voice Over (use this guide to get started)
  2. If you cannot reproduce the issue, email the 508 office contacts above to have them explain what they did to encounter the issue. You may even want to set up a meeting and screenshare to see them reproduce it.
  3. If you can reproduct the issue, you may consider doing a screen capture to illustrate any unexpected behavior, uploading it to YouTube (unlisted), and including the link in the github ticket you create. @BBarone1 and @rtluu are masters at this, so reach out to them for advice.
  4. Label the ticket with the appropriate product label and sprint team label. if the problem is in a header/nav/footer or other shared asset, label it sitewide
  5. Assign the ticket to the sprint team's product manager, or to @rroueche if they are sitewide

  6. Decide if the issue is launch-blocking and explain your reasoning in the ticket

    • if the feature is not usable on one of the listed screenreaders or with keyboard interaction, meaning the user cannot interact with it and get through the flow, then it must be fixed before launch
    • if the feature is hard to use or not ideally built but still usable with the listed screenreaders or with keyboard interaction, then it may be OK to launch, but the issue should be fixed in the following sprint
    • all other accessibility/508 issues should be fixed within the next program increment
    • there should not be open, unfixed a11y and 508 issues that linger for more than 3 sprints!

DSVA 508 Agreeement

This 2016 508 memo outlines an agreement between DSVA and the Section 508 office on the process for ensuring 508 compliance for DSVA-led products. This process likely differs from other VA IT projects because we launch new products and deploy far more frequently than is typical. Key things from the memo:

  • requires us to integrate our testing with a Deque product, either Attest or aXe. As of Feb 2018, we have been using the axe-core library in all of our end-to-end tests.
  • All code must pass those tests before deploying to production, which we have been practicing
  • The following aXe rule sets must be applied in our tests: section-508, wcag2a, wcag2aa which aim for both Section 508 compliance and to comply with the WCAG 2.0 Level AA standard.
  • Our automated tests are what permit us to not require 508 office evaluations to gate our launches. "This validation is in lieu of the current sign-off process because Digital Service is integrating automated scans and correction process into the continuous delivery deployment model." We thank our predecessors for this.
  • Periodic manual accessibility testing is required to catch any issues not found by automated testing. This should be done both by the vets.gov team and by the 508 office, as describe above in the Launch Criteria section.
  • Issues or false positives surfaced by manual testing (our own testing as well as that done by the 508 office) must be 1) documented, 2) categorized by severity. For "interim deployments" (read, minor updates), high severity defects must be fixed. For "final deployments" (read, major launches), all known defects (low, medium, high severity) must be fixed.
